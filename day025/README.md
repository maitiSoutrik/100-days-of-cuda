# Day 25: Parallel Point Cloud PassThrough Filter

## Introduction

This project implements a parallel PassThrough filter for point cloud data using CUDA. Point clouds, often generated by LIDAR or depth cameras, can be very large. Filtering is a common preprocessing step to remove irrelevant points or focus on a specific region of interest before tasks like object detection or registration. The PassThrough filter removes points that fall outside specified bounds along one or more axes (X, Y, Z). This operation is highly parallelizable, making it suitable for GPU acceleration.

This implementation focuses on the "PassThrough Filter" approach. It reads a point cloud from a simplified ASCII `.pcd` file, filters it on the GPU based on user-defined X and Y bounds (Z bounds are kept infinite in this example), and writes the filtered point cloud to a new `.pcd` file.

## Implementation Details

1.  **Point Representation:** A simple `struct Point { float x, y, z; }` is used.
2.  **PCD I/O:** Basic host-side functions (`readPCD`, `writePCD`) are included to handle simplified ASCII PCD files containing only XYZ data. These functions parse the header to find the number of points and read/write the `DATA ascii` section. Error handling is basic.
3.  **CUDA Kernels:**
    *   `passthrough_filter_kernel`: Each thread processes one input point. It checks if the point's coordinates (X, Y, Z) fall within the specified minimum and maximum bounds. A flag (1 for pass, 0 for fail) is written to a corresponding `d_flags` array.
    *   `compact_points_kernel`: This kernel performs a parallel compaction. Threads that processed points which passed the filter (flag == 1) use `atomicAdd` on a global counter (`d_out_count`) to determine the unique output index for their point. The point is then copied from the input array (`d_in_points`) to the correct position in the output array (`d_out_points`).
4.  **Host Code:**
    *   Parses command-line arguments for input/output filenames and filter bounds (min/max X, min/max Y). Allows `-inf` and `inf` for bounds.
    *   Reads the input PCD file.
    *   Allocates memory on the GPU for input points, flags, output points (sized pessimistically as the input), and the output counter.
    *   Copies input data to the GPU.
    *   Initializes flags and the output counter to zero using `cudaMemset`.
    *   Launches the `passthrough_filter_kernel`.
    *   Launches the `compact_points_kernel`.
    *   Uses CUDA events (`cudaEvent_t`) to measure the execution time of the two kernels.
    *   Copies the final count of filtered points (`h_out_count`) back from `d_out_count`.
    *   Allocates the correct size for the host output vector based on `h_out_count`.
    *   Copies the filtered points from `d_out_points` to the host output vector.
    *   Writes the filtered points to the output PCD file.
    *   Frees all allocated GPU memory.

## Key CUDA Features Used

*   **Kernel Launch:** Standard `<<<grid, block>>>` syntax.
*   **Grid/Block Indexing:** `blockIdx.x * blockDim.x + threadIdx.x` for thread ID calculation.
*   **Device Memory Management:** `cudaMalloc`, `cudaMemcpy`, `cudaMemset`, `cudaFree`.
*   **Atomic Operations:** `atomicAdd` for parallel compaction to safely determine output indices without race conditions.
*   **CUDA Error Handling:** `CHECK_CUDA_ERROR` macro for robust error checking.
*   **CUDA Events:** `cudaEventCreate`, `cudaEventRecord`, `cudaEventSynchronize`, `cudaEventElapsedTime` for performance measurement.

## Performance Considerations

*   **Parallelism:** The filtering and compaction steps are performed in parallel across potentially thousands of threads, significantly speeding up the process compared to a sequential CPU implementation for large point clouds.
*   **Atomic Operations:** While `atomicAdd` provides correctness for compaction, it can become a bottleneck if many threads contend for the same counter, especially on older architectures. For higher performance, a parallel prefix sum (scan) operation (like in Day 8) followed by a scatter write could be used, but `atomicAdd` is simpler for this demonstration.
*   **Memory Access:** Kernels access global memory. Performance depends on coalesced memory access patterns. The `Point` struct (3 floats) might lead to reasonably coalesced access.
*   **Overhead:** For very small point clouds, the overhead of memory transfers and kernel launches might negate the benefits of GPU parallelism.

## Building and Running

### Prerequisites

*   NVIDIA CUDA Toolkit (>= 10.0 recommended)
*   CMake (>= 3.10)
*   A C++ compiler (like g++) compatible with the CUDA Toolkit

### Building on Jetson Nano (or any Linux system with CUDA)

1.  **Clone the repository (if not already done):**
    ```bash
    git clone <your-repo-url>
    cd 100-days-of-cuda
    ```
2.  **Create a build directory:**
    ```bash
    mkdir build
    cd build
    ```
3.  **Run CMake:**
    ```bash
    cmake ..
    ```
4.  **Build the project (specifically Day 25):**
    ```bash
    make day025_passthrough_filter # Or just 'make' to build everything
    ```
    The executable will be located in `build/day025/passthrough_filter`.

### Running

1.  **Download a Sample PCD file:** You need an input PCD file. You can download sample files from the Point Cloud Library (PCL) data repository. Place the downloaded file on your Jetson Nano.
    *   **Recommended Sample (ASCII):** `table_scene_lms400.pcd`
        ```bash
        # On your Jetson Nano or local machine:
        wget https://raw.githubusercontent.com/PointCloudLibrary/data/master/tutorials/table_scene_lms400.pcd
        ```
    *   **Other Samples:** Explore the [PCL Data Repository](https://github.com/PointCloudLibrary/data) for more examples (note: some might be binary format or have different fields).

2.  **Execute the filter:** Provide the path to your downloaded PCD file, an output filename, and the filter bounds.
    ```bash
    ./build/day025/passthrough_filter <path_to_downloaded.pcd> <output_filename.pcd> <min_x> <max_x> <min_y> <max_y>
    ```
    *Example:* To keep points with X between 0.0 and 1.5, and Y between 0.0 and 1.2:
    ```bash
    ./build/day025/passthrough_filter ~/my_point_clouds/input.pcd filtered_cloud.pcd 0.0 1.5 0.0 1.2
    ```
    *Example using infinite bounds for Y:*
    ```bash
    ./build/day025/passthrough_filter ~/my_point_clouds/input.pcd filtered_cloud.pcd 0.0 1.5 -inf inf
    ```

## Execution Results / Output

The program will print:

1.  The input and output filenames and the filter bounds used.
2.  The number of points read from the input file.
3.  The kernel launch configuration (blocks and threads).
4.  The time taken for the GPU filtering and compaction steps in milliseconds.
5.  The number of points remaining after filtering.
6.  The number of points written to the output file.

*Example Console Output (using the sample input.pcd above and bounds X=[0, 1.5], Y=[-inf, inf]):*

## Learnings and Observations

*   The PassThrough filter is a fundamental operation easily mapped to the GPU's parallel architecture.
*   Atomic operations provide a simple way to implement parallel compaction, suitable for moderate point counts or when absolute peak performance isn't the primary goal.
*   Handling file I/O and command-line arguments remains host-side C++ code.
*   Error checking (`CHECK_CUDA_ERROR`) is crucial for debugging CUDA applications.
*   Performance measurement using CUDA events helps quantify the GPU acceleration benefit.
